# Phase 3 报告：交接文档质量提升

> 日期：2026-05-21 | 状态：✅ 完成

---

## 一、本阶段背景

Phase 2 生成的交接文档虽然包含了结构化信息，但经过质量验证，新人拿到文档后：
- **无法从零复现报表**：只有字段列表，没有开发步骤
- **不理解控件联动原因**：知道有哪些控件，不知道为何这样设计
- **数据集关系不明确**：不知道多个数据集如何关联，哪个是主数据集
- **字段业务含义缺失**：只有英文字段名，没有解释

核心问题归结为两点：
1. **解析器**只提取了"有什么"，没有提取标签-数据邻接关系和跨数据集共享字段
2. **LLM Prompt** 问的是"描述各部分"，没有要求解释"流向和原因"

---

## 二、改动清单

### 2.1 解析器增强（`parser/cpt_parser.py`）

新增两个语义提取功能：

| 功能 | 实现 | 作用 |
|------|------|------|
| `_extract_label_data_pairs()` | 扫描标签单元格与相邻数据绑定单元格的配对关系 | 还原表单字段的中文名称与数据集字段的对应关系 |
| `_find_shared_keys()` | 统计多数据集间的公共字段名 | 推断可能的 JOIN Key（如 studentno）|

新增 `CellBinding.expand_dir` 字段（row/col/None），以及 `summarize_to_dict()` 输出两个新字段：`label_data_pairs`、`dataset_shared_keys`。

### 2.2 LLM Prompt 重设（`agent/llm_analyzer.py`）

从"分章节描述"改为"以交互链路为核心"：

**新增输出字段：**

| 字段名 | 内容 |
|--------|------|
| `interaction_chains` | 每个控件的完整链路：控件作用 → SQL影响 → 数据展示 → **设计原因** |
| `dataset_relationships` | 主/辅数据集关系、JOIN 逻辑、数据流向 |
| `field_semantics` | 关键字段的业务含义（结合 label_data_pairs 推断）|
| `layout_description` | 表单型/列表型布局结构说明 |
| `development_steps` | 从零复现报表的可操作步骤列表 |
| `notes_or_risks` | 风险点（保留原有）|

**废弃字段：** `data_source_analysis`、`parameter_filter_logic`、`display_content_description`

**`_slim_for_llm()` 优化：**
- 共享字段只保留被 ≥3 个数据集共享的最多 10 条（避免噪音）
- 数据集超过 15 个时截断，优先保留实时查询数据集
- `max_tokens` 从 1024 提升到 3000
- 新增 `_repair_truncated_json()` 容错修复

### 2.3 交接文档结构重设（`agent/doc_generator.py`）

新八章结构：

| 章节 | 内容 |
|------|------|
| 一、报表概述 | 业务用途 + 布局结构 |
| 二、基本信息 | 文件基本属性 |
| 三、数据来源与字段关系 | DB连接 + 数据集 + 数据集关系（AI）+ 共享字段（解析）+ 字段语义（AI）|
| **四、控件→参数→数据→展示** | **每条交互链路的完整说明（核心新增）** |
| 五、单元格展示与字段对照 | 标签-数据配对 + 单元格绑定明细 |
| 六、从零复现开发步骤 | 可操作的复现步骤（AI 推断）|
| 七、注意事项与风险点 | AI 识别的风险 |
| 八、交接确认 | 人工填写模板 |

## 三、新版文档效果对比

| 报表 | v1 文档特征 | v2 文档特征 |
|------|------------|------------|
| 习题8 | 字段列表 + AI 未能推断第五节 | 6条表单配对 + 2条交互链路 + 13个复现步骤 |
| 制单报表 | 字段列表 + 参数描述段落 | 5条完整交互链路 + 8个复现步骤 |
| 出货汇总 | 字段列表 | 1条链路 + 15个复现步骤 |
| 客服报表 | （部分 section 未填充）| 11条交互链路（修复了 JSON 截断问题）|

## 四、卡点记录

| 卡点 | 现象 | 解决 |
|------|------|------|
| 客服报表有 80 个共享字段 | `_slim_for_llm` 传入过多数据 → LLM 输出被 token 截断 → JSON 解析失败 → `interaction_chains` 为空 | 共享字段过滤为 ≥3 个数据集共享的最多 10 条；max_tokens 提升到 3000；增加 `_repair_truncated_json` |
