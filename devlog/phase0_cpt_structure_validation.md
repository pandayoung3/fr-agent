## 二、XML 根节点结构

所有文件的根节点均为 `WorkBook`，包含版本信息：

```xml
<WorkBook xmlVersion="20211223" releaseVersion="11.5.0">
  <TableDataMap>      ← 数据集定义（核心）
  <ReportWebAttr>     ← Web 展示属性
  <Report>            ← 单 Sheet 的单元格布局
  <ReportParameterAttr> ← 参数面板（控件树）
  <StyleList>         ← 样式定义
  ...
</WorkBook>
```

一级子节点共约 15~17 个，核心解析节点为 4 个：`TableDataMap`、`Report`、`ReportParameterAttr`、`Connection`。

## 五、卡点深度评估

> 旧版本（8.x/9.x）兼容性不在 MVP 考虑范围内，已移除。以下对剩余三个卡点逐一评估其普遍性、实际影响和处置结论。

---

### 卡点 1：EmbeddedTableData 的 RowData 是 FR 私有编码

**普遍性评估**：普遍存在，三个文件均含 EmbeddedTableData，其中有内嵌数据的共 14 个数据集（出货汇总 5 个、客服报表 8 个、制单报表 1 个）。

**深入分析**：
- RowData 用于存储报表设计时静态缓存的数据，供预览和本地渲染使用
- 对于**交接理解场景**，RowData 的实际数据值意义有限——接手者需要理解的是"这张报表展示什么字段"，而非"缓存了多少行数据"
- 列名（`ColumnNames`）已完整可读，字段语义可从列名推断，EmbeddedTableData 的列名在三个文件中均成功提取
- 唯一损失：无法告知接手者"数据的实际范围或示例值"，例如下拉框里有哪些品牌选项

**结论**：**对 MVP 无实质阻塞，降级处理**。列名已足够支撑 LLM 做业务语义推断，RowData 解码不列入 Phase 1 任务。后续如需展示数据示例值，再单独研究编码格式。

---

### 卡点 2：动态 SQL 中的 `${}` 参数注入语法

**普遍性评估**：**非普遍，只影响含 DBTableData 的报表**。本次三个文件中仅"制单-1"有 SQL，客服报表和出货汇总全为 EmbeddedTableData，无 SQL，完全不受影响。在真实业务场景中，使用实时 DB 查询的报表比例估计在 30%~50%，该卡点属于"中频但非全局"。

**深入分析**：
- FR 的动态参数语法格式固定，均为 `${if(len(param)==0, "", "and field in ('" + param + "')")}` 这类模式
- 对于**静态分析**（交接文档生成），不需要执行 SQL，只需要：① 提取参数变量名，② 理解过滤逻辑
- 参数变量名可通过正则 `len\((\w+)\)` 和 `\+ (\w+) \+` 可靠提取，已在验证中确认
- LLM 可以直接理解 `${if(...)}` 语法并用自然语言解释其过滤逻辑

**样本验证**（制单-1 的 SQL）：
```sql
-- 原始（含 FR 动态参数）
SELECT * FROM `batch1`
where 1=1
${if(len(batch_number) == 0, "", "and batch_number in ('" + batch_number + "')")}
limit 10

-- 提取结论：参数变量 = batch_number，逻辑 = 当选择了批次号时过滤，否则查全量
```

**结论**：**非真正卡点，规则可解决**。用正则提取参数名 + 原始 SQL 文本直接交给 LLM 解释，无需预处理替换。Phase 1 中作为解析器增强项（低优先级），不阻塞 Agent 开发。

---

### 卡点 3：参数控件与数据集的联动关系

**普遍性评估**：**部分普遍，但实际情况远好于预期**。深入探查后发现，联动关系分两类，提取难度截然不同：

| 控件类型 | 代表 | 绑定结构 | 是否可提取 |
|---------|------|---------|-----------|
| 选项型（下拉/复选） | ComboBox / ComboCheckBox | `Dictionary → TableData → Name` | ✅ 已可提取，结构清晰 |
| 输入型（日期/文本/数值） | DateEditor / TextEditor / NumberEditor | 无数据集绑定（直接输入值） | 天然不需要，非问题 |

**实际提取率验证**：
- 客服报表（11 个有效控件）：8 个下拉型控件 100% 提取成功，3 个日期控件本不涉及数据集绑定
- 制单报表（13 个有效控件）：3 个下拉型控件提取成功，10 个日期/下拉控件未绑定数据集（属正常）

**已提取的绑定样本（客服报表）**：
```
ComboBox[Sid]         → 数据集[p_shop_c]       key=sid,        display=shop_name
ComboCheckBox[BrandId] → 数据集[r_brand_c]      key=brand_id,   display=brand_short_name
ComboCheckBox[SpuId]  → 数据集[p_spu_name]     key=spu_id,     display=spu_name
ComboCheckBox[GroupType] → 数据集[p_group_type] key=group_type, display=group_type
```

**真正未解决的部分（范围已大幅收窄）**：日期类/文本类控件的值如何作为 SQL 参数传入，需要在 SQL 中按参数变量名做匹配（卡点 2 的处理可覆盖这一场景）。

**结论**：**已大部分解决，残余问题可通过 SQL 参数名匹配覆盖**。将控件-数据集绑定提取（Dictionary 结构）升级为 Phase 1 解析器的正式能力，不再作为卡点管理。

**现有解析产出对 MVP 阶段（小样本跑通 Agent）已足够**。

## 八、结论

**Phase 0 验证通过**，解析器可行性已确认，三个文件全部成功解析，覆盖纯内嵌数据报表、实时 DB 查询报表、双 Sheet 报表三种典型场景。
