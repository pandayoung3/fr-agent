# 交接可用性评分标准

版本：P0 初版
状态：已确认

## 当前决策

评分标准列为 P0 最后一步，用于在进入 P1 前初步检验当前产品效果。

P0 只定评分标准，不实现自动评分功能。P1 再考虑将评分标准做成规则评分或 Agent 评分。

## 评分视角

评分要模拟一个真实交接 Agent：只拿 FR-Agent 解析、分析和导出的交接信息，尝试理解并复现原 CPT 报表。分数衡量“这些信息是否足够真实可用”，而不是单纯衡量文档是否好看。

## 100 分权重

| 模块 | 分值 | 评分重点 |
| --- | ---: | --- |
| CPT 结构解析 | 20 | 数据集、SQL、控件、单元格绑定、公式、父子格、条件高亮、填报配置是否完整可信 |
| 数据链路与复现信息 | 18 | 控件到参数、SQL、数据集、展示字段、填报字段的链路是否足够支撑复现 |
| 业务语义与指标解释 | 14 | 报表用途、字段语义、指标字典、公式含义和风险点是否具体、可核实 |
| 交接文档质量 | 13 | Markdown / HTML 是否结构完整、能指导接手人复现、明确人工核实点 |
| 前端交互体验 | 15 | 上传、进度、错误提示、结果 Tab、导出、重新分析是否清晰顺畅 |
| 问答可用性 | 8 | 追问是否基于当前报表上下文，是否能解释交互、公式和字段来源 |
| 安全与边界 | 7 | 是否避免读取业务数据、泄露密码、把 AI 推断写成事实 |
| 验收证据 | 5 | 是否有样例、命令、人工验收记录和已知缺口 |

## 模块评分锚点

### 1. CPT 结构解析：20 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 数据集与 SQL | 5 | 能列出主要数据集、类型、SQL、参数占位和数据源名称 |
| 参数控件 | 3 | 能识别控件名称、类型、绑定字段、取值来源和默认影响 |
| 单元格绑定 | 4 | 能定位主要展示字段、位置、数据集来源和扩展方向 |
| 公式与格式 | 3 | 能提取关键公式、格式类型和影响展示的计算逻辑 |
| 父子格与条件高亮 | 3 | 能说明主从展开关系、条件高亮规则和影响字段 |
| 填报配置 | 2 | 填报报表能识别表、主键、字段映射；查询报表该项可按不适用折算 |

### 2. 数据链路与复现信息：18 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 控件到 SQL 参数 | 5 | 能说明每个关键控件如何影响查询条件 |
| SQL 到数据集字段 | 4 | 能追踪 SQL 输出字段和数据集字段关系 |
| 数据集到展示字段 | 4 | 能说明页面关键字段来自哪个数据集和列 |
| 填报链路 | 2 | 填报报表能说明写回表、主键、字段映射和风险 |
| 复现步骤 | 3 | 接手人可按步骤重建查询、展示和交互主链路 |

### 3. 业务语义与指标解释：14 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 报表用途 | 3 | 能说明报表服务对象、业务场景和核心问题 |
| 字段语义 | 4 | 关键字段解释具体，不把字段名机械翻译成中文 |
| 指标字典 | 3 | 指标有来源字段、类型、单位、公式或口径说明 |
| 公式解释 | 2 | 关键公式有可核实解释，不只复述公式文本 |
| 风险点 | 2 | 能指出无法确认、依赖人工核实或潜在迁移风险 |

### 4. 交接文档质量：13 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 结构完整 | 3 | 包含概览、数据源、字段、交互、复现、风险和验收提示 |
| 可读性 | 2 | 接手人能快速找到关键结论，章节层级清晰 |
| 可复现性 | 4 | 文档能指导接手人重建报表主要功能 |
| 可核实性 | 3 | AI 推断和解析事实区分清楚，有人工核实提示 |
| 导出一致性 | 1 | Markdown / HTML 与页面分析结果一致 |

### 5. 前端交互体验：15 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 上传与状态反馈 | 4 | 用户能明确知道文件是否已上传、解析中、失败或完成 |
| 分析过程反馈 | 3 | SSE 进度、耗时提示和失败原因可读 |
| 结果组织 | 3 | 概览、链路、指标、步骤、问答、导出分区清晰 |
| 错误恢复 | 3 | 数据库增强失败、AI 失败、导出失败时有下一步动作 |
| 导出体验 | 2 | Markdown / HTML 下载命名清楚，内容与页面结果一致 |

### 6. 问答可用性：8 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 上下文绑定 | 3 | 回答明显基于当前 parsed + analysis，不泛泛而谈 |
| 交互解释 | 2 | 能解释参数控件、过滤条件和数据刷新影响 |
| 字段来源 | 1 | 能回答关键字段来自哪里、如何计算 |
| 风险表达 | 1 | 对不确定内容主动说明，避免编造 |
| 对话连续性 | 1 | 能利用最近上下文追问 |

### 7. 安全与边界：7 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 数据边界 | 2 | 只读取结构和元数据，不读取真实业务数据 |
| 密码处理 | 2 | 密码不进入日志、响应体、导出文档或 Git |
| AI 边界 | 2 | 明确 AI 输出需要人工核实，不把推断写成事实 |
| 失败降级 | 1 | DB / LLM 失败时不伪造结论 |

### 8. 验收证据：5 分

| 子项 | 分值 | 满分标准 |
| --- | ---: | --- |
| 样例记录 | 2 | 记录样例 CPT 类型、覆盖能力和验收人 |
| 命令记录 | 1 | 记录 build、lint、compile 或无法运行原因 |
| 缺口记录 | 1 | 扣分原因能进入需求池或 Handoff |
| 复验路径 | 1 | 后续修复后知道如何复测 |

## 封顶与否决规则

- 无法解析 CPT：总分最高 30。
- 无法完成 AI 分析且没有可读错误：总分最高 45。
- 导出文档不可用：总分最高 70。
- 前端主流程无法完成上传到导出：总分最高 60。
- 主查询链路、主要展示字段、关键交互三者里有两项无法复现：总分最高 65。
- 关键字段来源无法追踪：对应链路和语义模块至少扣 50%。
- 评分人无法根据输出复述报表用途和主链路：总分最高 65。
- 出现密码泄露、读取真实业务数据、AI 明显编造确定事实：直接判 D。

## 初步使用方式

1. 选择一份样例 CPT。
2. 使用 React + FastAPI 主线完成上传、解析、增强、分析、问答和导出。
3. 让评审者只看 FR-Agent 输出，尝试说明如何复现原报表。
4. 按 100 分标准逐项扣分。
5. 记录扣分原因，进入 P1 任务池。

## P0 评审记录模板

正式模板见 `docs/templates/SCORING_REVIEW.md`。每次样例 CPT 初评后，建议将扣分原因同步到 `docs/product/REQUIREMENTS_BACKLOG.md` 或 `docs/project/HANDOFF.md`。

## 建议评分等级

| 等级 | 分数 | 含义 |
| --- | ---: | --- |
| A | 90-100 | 可直接交付，只有轻微优化项 |
| B | 75-89 | 可用，但需要人工复核或少量修正 |
| C | 60-74 | 部分可用，存在明显缺口 |
| D | 0-59 | 不建议交付，需要返工 |

## P1 自动化候选

评分 MVP 可以先基于现有 `parsed` 和 `analysis` 做规则评分，不引入新模型：

- completeness：必要字段是否存在。
- confidence：关键链路是否能被解析证据支撑。
- risk：风险点数量和严重度。
- handoff_quality：导出文档章节是否完整。

## 暂不做

- P0 不实现评分功能。
- 不把评分当作绝对质量结论。
- 不用评分替代人工验收。
