# P1 前端可视化理解体验需求

## 背景

P0 已证明 FR-Agent 能完成上传、解析、AI 分析、问答、血缘和 Markdown / HTML 导出闭环。但当前体验仍偏“生成交接文档”，导出文件像主功能，而不是辅助能力。

P1 的产品重心应转向：让用户在 Web 界面里快速理解一张 CPT 报表，而不是先下载文档再阅读。

## 产品原则

- 导出 Markdown / HTML 是辅助能力，不是唯一或主要使用路径。
- 前端应优先帮助用户快速看懂 CPT 的结构、血缘、控件、字段、公式、风险和填报链路。
- 呈现方式应图文结合，优先使用图、结构化卡片、关系网络、hover / focus 解释和可展开详情。
- AI 分析应嵌入可视化节点和链路说明，而不是只作为长文本段落。
- 当前 P1 聚焦单个 CPT 的理解体验；批量 CPT 管理和知识库图谱放入 P2。

## P1 目标

把当前结果 Tab 升级为“报表理解工作台”：

- 用户上传 CPT 后，首先看到报表结构总览和关键风险。
- 用户可以用图谱方式查看控件、数据集、字段、公式、填报配置之间的关系。
- 用户 hover 或点击节点时看到解释、来源证据和 AI 备注。
- 用户可以在同一界面完成追问、定位字段、查看复现步骤和导出文档。

## P1 核心视图

### 1. 报表总览视图

目标：用一屏告诉用户“这张报表是什么、复杂度如何、最该关注什么”。

建议元素：

- 报表类型、FR 版本、数据集数、控件数、字段绑定数、公式数、写回配置。
- AI 一句话用途。
- 关键风险摘要。
- “理解路径”入口，例如：先看数据血缘、再看交互链路、再看填报配置。

### 2. 血缘图谱视图

目标：以图为主表达 CPT 关系。

建议节点：

- 参数控件。
- 数据集。
- SQL 参数或内嵌数据集来源。
- 展示字段。
- 公式单元格。
- 填报目标表和字段映射。
- 风险节点。

建议交互：

- hover 节点显示简短解释。
- 点击节点打开侧边详情面板。
- 高亮从控件到数据集到展示字段的路径。
- 区分数据来源、展示、计算、写回、风险等节点类型。
- 对无法匹配的控件或字段显示“需人工核实”状态。

### 3. 链路解释视图

目标：把“控件 -> 参数 -> 数据集 -> 字段 -> 展示 / 写回”说清楚。

建议方式：

- 每条链路以横向步骤流或泳道展示。
- 每一步都有证据字段，例如控件名、数据集名、字段名、单元格位置。
- AI 解释只作为辅助说明，必须和解析证据分开。

### 4. 字段与指标视图

目标：让用户快速找到字段业务含义和来源。

建议方式：

- 字段表支持按数据集、展示区域、是否写回、是否公式筛选。
- 字段点击后显示来源、展示位置、公式引用、AI 解释、人工核实提示。
- 指标字典和字段表可以合并为一个可筛选的信息视图。

### 5. 问答联动视图

目标：问答不只是聊天窗口，而是能联动当前图谱和字段。

P1 可以先做轻量版本：

- 问答入口保留。
- 回答中明确引用当前节点、字段、控件或链路名称。
- 后续 P2 再做“点击回答中的字段，高亮图谱节点”。

### 6. 导出视图

目标：保留交付能力，但不喧宾夺主。

建议定位：

- 导出放在“交付 / 分享”区域。
- Markdown / HTML 是对当前工作台理解结果的快照。
- 导出前提示：AI 推断需人工复核。

## P2 延展：CPT 知识图谱

当进入批量 CPT 解析和报表资产管理后，可以借鉴 Obsidian graph 的方式：

- 一个节点代表一张 CPT。
- 数据集、数据库表、字段、指标可以成为跨报表共享节点。
- 报表之间通过共同数据源、字段、指标、写回表建立关系。
- 支持客户的 CPT 报表管理、数据资产管理、影响分析和迁移评估。

该方向放入 P2，不阻塞 P1 单报表体验优化。

## P1 验收标准

- 上传单个 CPT 后，用户无需导出文档，也能在 Web UI 中说清楚报表用途、数据来源、关键控件、展示字段、公式和风险。
- 血缘图谱可交互，hover / click 至少能查看节点解释。
- 导出入口仍可用，但不再是结果页的唯一主线。
- AI 文本、解析证据、人工核实提示在界面上有清晰边界。
- 新 UI 通过 `npm run lint` 和 `npm run build`。
